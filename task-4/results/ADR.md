# ADR-4: Использование Spring Batch для локального ETL POC по обогащению товарных данных
# Статус: принято

## 1. Контекст

Нужно проверить, подходит ли Spring Batch как прикладное решение для ETL-пайплайна, который:

- считывает исходные товары из `product-data.csv`;
- загружает справочник программы лояльности из `loyality_data.csv` в PostgreSQL;
- обогащает товары данными о лояльности;
- сохраняет итоговые данные в таблицу `products`;
- запускается локально через `docker compose` и может быть перенесён в облачную среду.

Ограничения и ожидания:

- POC должен быть воспроизводим локально без ручной сборки образа.
- Результат должен быть проверяемым через логи и содержимое БД.
- Решение должно поддерживать повторный запуск без ручной очистки таблиц.
- Архитектура должна оставаться простой для MVP, но не блокировать перенос в Kubernetes/CronJob.

## 2. Драйверы решения

Ключевые требования:

- пакетная обработка и контроль шагов job;
- хранение метаданных выполнения job и прозрачная диагностика сбоев;
- простая интеграция с PostgreSQL;
- минимальный объём платформенной обвязки для локального демо;
- идемпотентность повторных запусков;
- понятный путь к industrialized deployment в облаке.

## 3. Решение

Выбран **Spring Batch** в виде отдельного batch-приложения, запускаемого в контейнере рядом с PostgreSQL.

POC построен как job `importProductJob` из двух шагов:

1. `loadLoyaltyStep`
   Загружает `loyality_data.csv` в таблицу `loyality_data` с `upsert`, чтобы справочник можно было обновлять повторным запуском.
2. `importProductsStep`
   Читает `product-data.csv`, через `ProductItemProcessor` подменяет `productData` значением из `loyality_data`, если запись найдена, и сохраняет результат в `products` через `upsert`.

Технические решения внутри POC:

- схема БД инициализируется автоматически через `schema-all.sql`;
- Spring Batch сам создаёт свои служебные таблицы метаданных;
- контейнер приложения собирается multi-stage Dockerfile, поэтому локально достаточно `docker compose up --build`;
- `postgresdb` имеет healthcheck, а batch-контейнер стартует только после готовности БД;
- для lookup в процессоре используется параметризованный SQL, без конкатенации строки запроса;
- итоговая таблица заполняется идемпотентно через `ON CONFLICT DO UPDATE`.

## 4. Почему это решение подходит

Плюсы выбора Spring Batch для этого кейса:

- модель `reader -> processor -> writer` прямо соответствует постановке задачи;
- есть встроенные job/step semantics, retryability и metadata tables;
- достаточно лёгкий стек для MVP, без отдельного orchestration layer;
- интеграция с JDBC/PostgreSQL тривиальна;
- решение легко переносится в облако как одноразовый Job/CronJob.

Ограничения выбора:

- Spring Batch хорош для bounded batch ETL, но не для near real-time стриминга;
- оркестрация сложных межсервисных DAG остаётся слабее, чем у Airflow;
- при росте числа источников и сценариев может потребоваться вынос планирования и observability наружу.

## 5. Альтернативы

### Альтернатива 1. Кастомный Spring Boot/JDBC ETL без Spring Batch

Плюсы:

- меньше абстракций;
- быстрее собрать самый простой demo.

Минусы:

- пришлось бы вручную реализовывать шаги, restartability, метаданные и контроль прогресса;
- решение хуже масштабируется по сложности ETL.

Вердикт:

Для одноразового скрипта подходит, для демонстрации технологического выбора под ETL-платформу хуже Spring Batch.

### Альтернатива 2. Apache Airflow или Prefect

Плюсы:

- сильная оркестрация и scheduling;
- хорошо подходят для множества зависимых пайплайнов.

Минусы:

- для текущего локального POC это лишняя платформенная сложность;
- нужен отдельный orchestration stack поверх самого ETL-кода.

Вердикт:

Подходит как следующий шаг при росте количества пайплайнов, но избыточно для текущей задачи.

### Альтернатива 3. Kafka Connect / stream processing

Плюсы:

- хорошо работают для потоковой интеграции и near real-time событий.

Минусы:

- исходный кейс основан на CSV и пакетной обработке, а не на событиях;
- эксплуатационная сложность выше, чем ценность для POC.

Вердикт:

Не соответствует характеру задачи.

## 6. Последствия выбора

Позитивные:

- получен воспроизводимый локальный ETL POC;
- путь миграции в облако очевиден: контейнер + external PostgreSQL + scheduler;
- поведение job можно диагностировать по логам и служебным таблицам Batch.

Негативные:

- batch-приложение выполняется как отдельный одноразовый контейнер;
- для production потребуется вынести секреты, метрики, алерты и расписание.

## 7. Риски и меры

- Риск: изменение формата CSV.
  Мера: фиксированные reader-конфигурации и контракт на заголовки/порядок колонок.

- Риск: повторный запуск приводит к дублям.
  Мера: `upsert` в `loyality_data` и `products`.

- Риск: БД не готова к моменту старта job.
  Мера: `healthcheck` PostgreSQL и `depends_on.condition=service_healthy`.

- Риск: решение используют как realtime ETL.
  Мера: явно зафиксировать в ADR, что это batch-only POC.

## 8. Итог

Для данного кейса **Spring Batch выбран как базовая технология ETL POC**: он покрывает пакетную обработку CSV, обогащение через PostgreSQL, проверяемость результата и даёт понятный путь к облачному развёртыванию без чрезмерной платформенной сложности.
