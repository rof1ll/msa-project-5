# ADR-5: Обоснование мониторинга, логирования и оповещений
# Статус: принято

## Контекст

Для Spring Batch-приложения нужно локально развернуть наблюдаемость: метрики, централизованные логи и базовые оповещения. Решение должно быть простым для `docker compose`, но при этом оставаться пригодным для переноса в облако.

## Выбранное решение

Использован стек:

- **Prometheus** — сбор метрик;
- **JMX Exporter** — экспорт JMX-метрик приложения;
- **Grafana** — dashboard;
- **Filebeat + Logstash + Elasticsearch + Kibana** — централизованные логи;
- **Prometheus alert rules** — оповещения.

## Почему так

- **JMX Exporter** подходит под задание и позволяет отдавать метрики без прямой зависимости приложения от Prometheus.
- **Prometheus + Grafana** — стандартный и понятный стек для мониторинга и визуализации.
- **ELK + Filebeat** позволяют собирать и искать структурированные JSON-логи без прямой отправки логов из приложения в Elasticsearch.
- **Prometheus rules** проще хранить и проверять как код, чем настраивать alerts только через UI.

## Что собираем

Метрики:

- доступность exporter (`up`);
- JVM-метрики;
- batch-метрики последнего запуска:
  `batch_job_last_status_code`,
  `batch_job_last_duration_millis`,
  `batch_job_last_read_count`,
  `batch_job_last_write_count`,
  `batch_job_last_skip_count`.

Логи:

- приложение пишет JSON-логи в файл;
- Filebeat читает файл из общего volume;
- Logstash отправляет события в Elasticsearch;
- Kibana используется для поиска и анализа.

Alerts:

- `BatchJobFailed` — последний запуск завершился ошибкой;
- `BatchJobProcessedNoRows` — job завершился, но ничего не записал.

## Итог

Выбранный стек закрывает требования задания по мониторингу, логированию и оповещениям, остаётся понятным для локального POC и не привязывает batch-приложение напрямую к observability-инфраструктуре.
