# Выгрузка B2B прайс-листов

## Сравнение вариантов

| Критерий | Spring Batch | Apache Airflow | K8s CronJob | Spark |
| --- | --- | --- | --- | --- |
| Наличие конфигурации CRON-расписания | Частично: нужен внешний scheduler или запуск через Kubernetes | Да, встроено | Да, встроено | Нет, нужен внешний scheduler |
| Сложность реализации логики обработки | Средняя: сильный framework, но для простой выгрузки тяжеловат | Средняя: DAG удобен, но здесь избыточен | Низкая: достаточно одного контейнера и одного job-сценария | Высокая: слишком тяжёлый стек для простого экспорта |
| Ресурсоёмкость решения | Низкая или средняя | Высокая: scheduler, webserver, metadata DB, workers | Низкая | Очень высокая |
| Масштабируемость и сложность реализации | Хорошая, но оправдана при сложных batch-пайплайнах | Высокая как orchestrator, но для одной задачи не окупается | Достаточная для текущей нагрузки, масштабируется ресурсами pod/job | Очень высокая, но не нужна при таком объёме |
| Развёртывание в облаке и интеграция с микросервисами | Средняя | Средняя или высокая | Низкая: нативно для Kubernetes | Высокая |
| Логирование и мониторинг | Хорошо через приложение и Actuator/Prometheus | Хорошо, но с отдельным контуром эксплуатации | Хорошо: stdout, metrics endpoint, Kubernetes events, Prometheus | Хорошо, но дорого по операционной сложности |

## Выбранное решение

Для этой задачи оптимален `Kubernetes CronJob` с отдельным контейнеризированным сервисом `exporter`.

Причина выбора простая:

- задача запускается по расписанию один раз в день;
- объём данных небольшой: `5 000-10 000` строк на выгрузку;
- дополнительной бизнес-обработки почти нет, нужен в основном `SELECT + JOIN + запись в файл`;
- система уже живёт в облачной микросервисной среде, а значит контейнер и `CronJob` хорошо вписываются в текущую эксплуатационную модель;
- `Airflow` и `Spark` для такого сценария избыточны, а `Spring Batch` полезен при более сложных batch-сценариях, чем здесь.

## Вывод

Для регулярной ежедневной выгрузки, где нет сложной обработки данных, достаточно Kubernetes CronJob и небольшого экспортного сервиса или простого Spring Batch Job, запускаемого по расписанию. Spark для такого сценария избыточен. Airflow имеет смысл использовать только если он уже принят в системе как общий оркестратор для других пайплайнов, как в задании 1.

В качестве целевого решения для To Be выбран K8s CronJob с контейнером, который выполняет экспорт из SQL в CSV. Это самый простой и естественный вариант для облачной микросервисной среды: он хорошо интегрируется в существующую инфраструктуру и обеспечивает предсказуемый запуск по расписанию. Если же в компании основной стек построен вокруг Java, разумной альтернативой будет Spring Batch с одним шагом, запускаемым по cron.
